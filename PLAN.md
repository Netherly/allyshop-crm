# План разработки CRM (allyshop-crm)

CRM для учёта одежды: товары/SKU, склад через движения, наборы/ростовки, клиенты,
заказы, оплаты, возвраты, доставка Новой Почтой, финансы и аудит.

Источник требований: `TZ_CRM_odezhda_vibecoder.docx` (извлечённый текст — `tz_text.txt`).

> Этот файл — живой чек-лист. По мере выполнения отмечаем пункты `[x]` и фиксируем,
> что осталось. Разработка идёт поэтапно: следующий этап начинаем только после
> рабочего предыдущего.

## Пожелания заказчика

### Внешний вид (UI/UX)

- [ ] Стилизация интерфейса — **как у KeyCRM** (keycrm.app): чистый светлый интерфейс,
      левое боковое меню разделов, верхняя панель, основной контент — таблицы и карточки,
      аккуратные отступы, спокойная цветовая палитра с одним акцентным цветом, статусы
      выделяются цветными бейджами. Ориентируемся на этот визуальный язык во всех разделах.

### Стиль кода и комментариев

- Писать **только базовые комментарии** — коротко, что делает функция/блок.
- Комментарии в стиле разработчика, а НЕ «ИИ-комментарии»: без воды, без пояснения
  очевидного, без длинных вступлений вроде «This function is responsible for…».
- Не комментировать каждую строку. Комментарий — там, где он реально помогает понять
  намерение или неочевидную логику (например, складские формулы, списание/возврат).
- Язык комментариев — русский (как и интерфейс).

### Тестирование (юнит-тесты)

- Пишем юнит-тесты по ходу разработки, чтобы ловить баги раньше — не откладываем всё на конец.
- Инструменты: **Vitest** (backend и frontend), **Supertest** для эндпоинтов API,
  **Testing Library** для ключевых компонентов UI.
- Обязательно покрывать тестами **критичную бизнес-логику** (там цена ошибки выше всего):
  расчёт остатка по движениям, доступность набора, разложение набора на товары,
  списание/возврат заказа (включая запрет двойного списания/возврата), пересчёт
  `payment_status`, фиксацию цены/себестоимости на момент продажи.
- Тесты бизнес-логики — на чистых функциях/сервисах, без обращения к реальной БД, где это
  возможно (быстро и стабильно). Для API — интеграционные тесты на тестовой БД.
- Команда `npm test` в backend и frontend; зелёные тесты — условие готовности этапа.

## Сквозные правила (требования к AI-кодеру, §20 ТЗ)

Эти правила обязательны на КАЖДОМ этапе, не выносятся в отдельную задачу:

- [ ] Остаток считается ТОЛЬКО по движениям (`stock_movements`), не одним полем в товаре.
- [ ] Пароли — только bcrypt-хэш, никогда не хранить открытым текстом.
- [ ] Заказ не списывается дважды (`stock_written_off`) и не возвращается дважды (`stock_returned`).
- [ ] Старые заказы не пересчитываются при изменении цены товара — цена и себестоимость
      фиксируются в строке заказа на момент продажи.
- [ ] Набор НЕ является складским товаром — остаток считается по товарам внутри.
- [ ] Перед списанием проверять остатки всех товаров внутри набора.
- [ ] Доставка хранится отдельно от клиента (`order_delivery`).
- [ ] Архивирование (`is_active=false`) вместо физического удаления важных данных.
- [ ] Критичные действия логируются в `audit_log`.
- [ ] Складские операции списания/возврата — в транзакции БД (атомарно).
